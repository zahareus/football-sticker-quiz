Синхронність WEB+IMAGE → це системна проблема крауйлу/видачі, а не контенту окремих сторінок.

### Збіг у часі (git)
- **12.05 09:08-14:11** — масова регенерація CWV-фіксу: index + 697 клубів + 3422 стікери + sitemaps (6 комітів).
- **16.05 12:40-14:00** — десятки авто-генерацій + auto-reconcile orphaned pages.
- Обвал стартує 14-15.05 — одразу після масової регенерації 12.05.

### Сторінки здорові — проблема не в них
- Homepage / stickers/3201 / clubs/1095: HTTP 200, canonical правильний, без noindex, INDEXING_ALLOWED, Rich Results PASS (Breadcrumbs).
- robots.txt: `search=yes`, Googlebot дозволений (блокуються лише AI-боти).
- Усі індексовані («Submitted and indexed»).

### Реальний симптом — крауйл заморожено
- `stickers/3201.html`: last crawled **2026-05-11** (29 днів тому).
- `clubs/1095.html`: last crawled **2026-04-05** (66 днів тому).
- Усі sub-sitemaps мають `lastmod 2026-06-06` — тобто кожна регенерація піднімає lastmod
  усім сторінкам одразу. Часті повні регенерації з «усе змінилось сьогодні» отруюють
  crawl-budget: Google перестає довіряти lastmod і зменшує частоту крауйлу.

**Гіпотеза root cause:** lastmod-churn від частих масових регенерацій (12.05 + щоденні авто-генерації)
+ ймовірне знецінення «змінених, але по суті ідентичних» сторінок → Google зарізав crawl-budget.

## Рекомендації (за пріоритетом)
1. **🔴 Розморозити crawl-budget.** Припинити бамп lastmod усім сторінкам на кожній регенерації —
   lastmod має мінятись ТІЛЬКИ для реально змінених сторінок. Авто-генерація однієї нової картки
   не повинна переписувати sitemap-lastmod 3500 стікерів.
2. **🔴 Request indexing** для топ-сторінок (home, catalogue, флагмани) вручну в GSC, щоб
   «розбудити» Googlebot.
3. Закрити `sitemap-images.xml` 404 (P3) — реанімувати image search.
4. Дочекатись трафіку для CrUX field data, тоді гейтити LCP реально.

## ✅ Зроблено 10.06 (фікс root cause)

**Commit `ce05eb6cf` (main, deployed) — `scripts/generate-sitemaps.js`**

Було: `lastmod = today` для КОЖНОГО URL і кожного sub-sitemap на кожному запуску
(`generate-sitemaps.js:62`). Кожна авто-генерація однієї картки переписувала дату 4500+ сторінкам.

Стало: lastmod читається з уже закоміченого sitemap і **зберігається per-URL**; `today`
отримують лише реально нові URL. lastmod індексу = max дата серед URL у sub-sitemap.
CI-safe — читає закомічений XML, а не файлові mtime (їх скидає `git checkout` у CI).
Escape-hatch `SITEMAP_TOUCH_ALL=1` для свідомих site-wide змін шаблону.

**Артефакти перевірки:**
- Регенерація з чистого baseline → **0 diff** (ідемпотентно, churn усунено).
- `SITEMAP_TOUCH_ALL=1` → 4516 змінених lastmod (механізм працює), після `git restore` + звичайного прогону знову 0 diff.
- `npm test` 56/56 ✅, `npm run test:generators` 57/57 ✅.

**Ефект:** наступні авто-генерації більше не отруюють crawl-budget. Поточні живі sitemaps
лишаються на `lastmod 2026-06-06` (без подальшого churn). Очікуємо поступове відновлення
частоти крауйлу і показів. Контроль — наступна перевірка через 2 тижні (~24.06).

**Не закрито (окремо):** `sitemap-images.xml` 404 (P3), ручний request-indexing у GSC,
ізоляційний борг clubs/1095. LCP/Perf — чекаємо CrUX field data (трафіку замало).
